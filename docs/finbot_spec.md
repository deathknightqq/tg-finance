# FinBot — Telegram-бот финансового учёта поверх выписок Kaspi

Спека утверждена по методологии: бриф → ресёрч → концепции → скелет. Этот файл — результат шагов 0-3. Клод Код исполняет шаг 4 (сборка чанками с чекпоинтами) и шаг 5 (self-review). Полировка — только после.

## 1. Контекст и цель

Личный проект на двоих юзеров (автор + партнёрша), не SaaS. Цель — рабочий инструмент, которым автор реально пользуется 3+ месяца, и вау-демо для друзей.

Ключевой инсайт продукта: аналитика в приложении Kaspi видит только покупки (по MCC), а переводы физлицам для неё серая масса — при том что у автора переводов в 2.6 раза больше покупок по объёму. Плюс через карту автора транзитом ходят чужие/рабочие деньги, поэтому сырая выписка врёт о личных финансах. Бот строит честную картину: категоризирует переводы, отделяет транзит, опционально объединяет пару.

Почему без API: у Kaspi нет публичного API, open banking KZ недоступен физлицам. Вход данных — только PDF-выписка, которую юзер сам скидывает боту (раз в неделю/месяц). Push-уведомления форвардить технически невозможно — не предлагать.

## 2. Скоуп MVP (итерация 1)

Входит: Kaspi Gold PDF (локаль RU обязательно; KZ/EN — см. риски), соло-режим как дефолт, обучаемая категоризация с опросником пачками, разметка принадлежности (моё/транзит/внутрисемейное), обработка пар списание+возврат, ручной бюджет месяца, отчёты, опциональная связка двух юзеров с выборочным шарингом и матчингом взаимных переводов.

Осознанно НЕ входит (не пилить, даже если просится): Halyk и другие банки, LLM в любом виде, прогноз кэшфлоу сложнее «бюджет минус потрачено», больше двух юзеров/группы, веб-интерфейс, деплой на сервер (только локалхост + Docker).

## 3. Стек

Python 3.12, aiogram 3.x (long polling, без вебхуков), pdfplumber, SQLite + SQLAlchemy 2.x (переезд на Postgres потом должен быть заменой connection string), pytest, Docker + docker-compose. Токен бота и настройки — через .env, в git не коммитить.

## 4. Модель данных

- `users`: id, tg_id, name, couple_id (nullable — соло-режим полноценный)
- `statements`: id, user_id, period_start, period_end, opening_balance, closing_balance, file_hash, uploaded_at. Сырой PDF после парсинга УДАЛЯЕТСЯ. ФИО, ИИН, IBAN, номер карты — не сохраняются нигде и никогда.
- `transactions`: id, statement_id, user_id, date, amount (signed, тиыны как integer — не float), op_type (purchase/topup/transfer/withdrawal), counterparty_raw, counterparty_id (nullable), category_id (nullable), ownership (mine/transit/intrafamily/unassigned), matched_tx_id (nullable, ссылка на транзакцию партнёра), netted_with_id (nullable, ссылка на парную операцию возврата), currency_note (nullable, для строк типа «(- 23,20 USD)»)
- `counterparties`: id, scope (couple_id или user_id), name_normalized, category_id, default_ownership, netting_rule (nullable: collapse/show). Сердце системы: сюда пишутся ответы юзера, повторный вопрос про того же контрагента — баг.
- `categories`: id, user_id (nullable для сид-набора), name, is_shared (флаг видимости партнёру). Сид: продукты, кафе и доставка, транспорт, подписки, связь, аптека, коммуналка, дом/быт, наличка, транзит, семья, развлечения, прочее.
- `budgets`: user_id, month, amount — руками через команду.
- `question_queue`: user_id, counterparty_id, weight (сумма × частота), status.

## 5. Пайплайн обработки выписки

1. Приём PDF → pdfplumber → извлечение шапки (период, opening/closing) и таблицы транзакций.
2. Склейка continuation-строк: строка без даты (например `(- 23,20 USD)`) приклеивается к предыдущей транзакции как currency_note. Разрывы таблицы между страницами не должны терять и не должны дублировать строки.
3. Нормализация: суммы «- 1 800,00 ₸» → signed int в тиынах (пробелы-разделители, запятая-десятичная); даты DD.MM.YY → date.
4. **Golden Rule (обязательный гейт):** opening + Σ(все транзакции) == closing, копейка в копейку, сверка со сводным блоком выписки. Не сошлось → выписка отклоняется целиком с отчётом (сколько распарсили, какая дельта). Молча принимать битый парс запрещено.
5. Дедуп при перекрытии периодов: сравнение мультимножеств ключей (date, amount, op_type, counterparty_raw) с учётом кратности. Три одинаковых Avtobys по 110 ₸ в день — норма, не дубли. Хэш всего файла — только против повторной загрузки того же PDF.
6. Детект пар списание+возврат: одинаковая сумма, тот же counterparty_raw, окно ±2 дня, противоположные знаки. Поведение — вопрос юзеру при ПЕРВОЙ встрече пары по этому контрагенту («Схлопнуть и не считать в статистике / показывать отдельной строкой?»), ответ сохраняется в netting_rule и больше не спрашивается.
7. Авторазметка: сначала counterparties (обученное), затем сид-правила по подстрокам (YANDEX.GO/EDA/LAVKA/PLUS/ZAPRAVKI, WOLT, Avtobys → транспорт, Beeline, APPLE.COM, Google One, слова «аптека»/«фарм», «Банкомат»/«Снятие» → наличка, «ЕРЦ» → коммуналка и т.п.). Неопознанное → question_queue со статусом unassigned; отчёты при этом НЕ блокируются, неразобранное показывается отдельной строкой.
8. Опросник: после разбора бот сообщает «нашёл N неизвестных» и показывает 5 самых весомых (weight = Σ|сумма| × частота). Формат вопроса: контрагент, примеры операций, инлайн-кнопки категорий + «транзит» + «своя категория». После пачки — кнопки «ещё 5» / «хватит, потом». Команда /unsorted открывает очередь в любой момент.

## 6. Продуктовые решения (зафиксированы юзером, не менять)

- Пара опциональна, соло — дефолт. Связка по инвайт-коду.
- Шаринг выборочный: флаг is_shared на категории. Партнёр видит только расшаренные категории (итоги и операции внутри них). Нерасшаренное не видно ни в каком виде.
- Матчинг взаимных переводов пары: встречные знаки, равная сумма, дата ±1 день, взаимные контрагенты. Полное совпадение — авто-пометка intrafamily (исключается из статистики обоих). Частичное — вопрос с кнопкой, молча не схлопывать.
- «Свободно до конца месяца» = бюджет месяца (введён руками, /budget) − потрачено с ownership=mine. Автодетект регулярных списаний остаётся как информер («найдены регулярные: ANTHROPIC ~11k/мес, ЕРЦ ~34k/мес...»), но в расчёт свободного не вмешивается.

## 7. Чанки сборки и чекпоинты

Собирать строго по одному чанку. После каждого — стоп и ответ на чекпоинт-вопрос. Не переходить дальше, пока чекпоинт не пройден на реальных тестовых PDF.

1. **Парсер** (чистый модуль без бота): PDF → список транзакций + шапка. Чекпоинт: эталонная выписка даёт ровно N транзакций, Golden Rule сходится в ноль, continuation-строки приклеены.
2. **База + ingest + дедуп**: тот же файл дважды → 0 новых записей; перекрывающиеся недельная и месячная выписки → без потерь и без дублей.
3. **Бот-каркас**: /start, приём PDF-документа, прогон пайплайна, ответ «разобрано N, неизвестных M». Чекпоинт: цикл работает на локалхосте end-to-end.
4. **Категоризация + опросник**: сид-правила, словарь, пачки по 5, /unsorted. Чекпоинт: вторая загрузка выписки того же юзера задаёт ≤5 новых вопросов.
5. **Принадлежность + netting**: транзит, вопросы по парам возвратов, netting_rule. Чекпоинт: месячная статистика автора не содержит транзитных 1.6M и рабочих тест-пар, итог «моих» расходов выглядит правдоподобно.
6. **Отчёты + бюджет**: /report месяц/неделя по категориям, /budget, «свободно до конца месяца». Чекпоинт: цифры отчёта сходятся с ручным пересчётом по базе.
7. **Пара**: инвайт, is_shared, матчинг взаимных переводов, совместный отчёт. Чекпоинт: на выписках обоих юзеров взаимные переводы схлопнуты, приватные категории не видны партнёру.

После чанка 7 — отдельная сессия self-review в роли критика, не автора: пройтись по коду и поведению против этой спеки, составить список расхождений и хрупких мест, только потом фиксить. Полировка (тексты, красота отчётов) — последней.

## 8. Тесты

Золотой набор: обезличенные реальные выписки (месячная RU — уже есть; казахская, английская, за длинный период 3-6 мес, файл второго юзера — добудет автор). Парсер обязан либо корректно разобрать, либо явно отказать с понятной ошибкой («локаль не поддерживается»), но не выдать мусор — Golden Rule это гарантирует. Юнит-тесты на: нормализацию сумм/дат, склейку continuation, дедуп мультимножеством, детект пар возвратов, матчинг пары, расчёт отчёта.

## 9. Явные запреты

- Не хранить сырые PDF после парсинга; не сохранять ФИО/ИИН/IBAN/номер карты.
- Не тащить LLM, внешние API категоризации, MCC-базы — ядро детерминированное.
- Не пилить несколько чанков одним заходом без чекпоинтов.
- Не принимать выписку с несошедшимся Golden Rule «примерно».
- Не расширять скоуп из раздела 2 без явного решения автора.
- Токены и данные — только локально, ничего не отправлять наружу.

## 10. Известные риски

- Локали KZ/EN: ключевые слова шапки и типов операций отличаются. Парсер должен детектить язык по маркерам («ВЫПИСКА» / «ҮЗІНДІ КӨШІРМЕ» / «STATEMENT») и либо парсить, либо честно отказывать. Приоритет поддержки: RU обязателен, KZ желателен (партнёрша может пользоваться казахской локалью), EN опционален.
- Формат Kaspi стабилен годами, но парсер держать изолированным модулем с интерфейсом «PDF → транзакции», чтобы смена формата или добавление Halyk (итерация 2) было новым адаптером, а не переписыванием.
- Иконка «сумма заблокирована» (pending) не попадает в текстовый слой — pending-операции в тексте неотличимы от обычных. Принято: не детектим, пары возвратов ловим неттингом из п.5.6.
