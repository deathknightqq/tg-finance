# FinBot — self-review (роль критика, спека §7)

Дата: 2026-07-21. Все 7 чанков собраны, 111 тестов зелёные. Это ревизия
против спеки и здравого смысла: список расхождений, узких мест и хрупкостей.
Порядок — по приоритету (impact × вероятность). Фиксить сверху вниз.

## A. Корректность — чинить в первую очередь

1. **Дубликат «своей категории».** `apply_answer` при `custom_name` всегда
   создаёт новую `Category` без проверки существующей
   ([categorize.py:458](../finbot/categorize.py)). Ответишь «калым» для двух
   разных контрагентов — получишь две категории «калым», и в `/report` они
   встанут двумя строками. Нужен lookup-or-create по имени (case-insensitive)
   в рамках юзера/сид-набора.

2. **Залипание FSM «своя категория».** Клик «✏️ своя категория» ставит
   state `waiting_name`, а сбрасывает его только ввод текста. Если между этим
   прислать PDF, нажать кнопку на другой карточке или вызвать /report —
   state остаётся, и следующий обычный текст съедается как название категории
   для старого вопроса ([bot.py](../finbot/bot.py), хендлер `cb_custom`).
   Нужны: кнопка «отмена», сброс state при других действиях, TTL.

3. **Пустое имя кастомной категории.** `custom_name.strip()` от пустой строки
   создаёт категорию без имени. Валидация: непусто, разумная длина, не
   дублирует сид.

## B. Продукт/UX — высокая ценность для «3+ месяцев реального использования»

4. **Нельзя исправить ответ.** Раскатегоризовал контрагента неверно — отката
   нет. За месяцы реального использования мискликов накапливается, и это
   ГЛАВНАЯ причина, по которой такие трекеры забрасывают. Нужна команда
   `/counterparty <имя>` или кнопка «изменить категорию» в отчёте.

5. **Только текущий месяц / 7 дней в отчёте.** Твой же запрос «расходы до
   10-го» не покрыт: нет произвольного периода. Нужен `/report июнь`,
   `/report 01.06-10.06` или инлайн-переключалка месяцев.

6. **Категориями нельзя управлять.** Нет rename/merge/delete. Исследование
   трекеров прямо предупреждает: разрастание категорий и «прочее» как чёрная
   дыра убивают трекер. Нужен `/categories` с редактированием.

7. **Переводы между своими счетами (классический double-counting).** Главная
   ошибка всех трекеров: движение денег между своими счетами считается
   расходом/доходом. Если гоняешь деньги на Kaspi Депозит и назад — мы это
   посчитаем. Kaspi в сводке отделяет «Переводы на свои счета» / «Поступления
   со своих счетов», но в строках операций этого нет. Сейчас спасает только
   ручная пометка «транзит». Стоит: явная категория «свои счета» + подсказка.

## C. Сохранность данных — то, что реально пугает

8. **Нет бэкапа `finbot.db`.** Файл (правильно) в .gitignore — там перс.
   данные. Но если он побьётся или диск умрёт, три месяца твоей ручной
   разметки исчезнут безвозвратно. Все посмертия трекеров — про потерю данных.
   Нужно: периодическая копия + команда `/export` (CSV/JSON выгрузка).

## D. Операционная хрупкость

9. **Бот надо руками перезапускать каждую сессию** — умирает с терминалом,
   нет single-instance-гарда, нет автозапуска. Спека разрешает только
   localhost + Docker, поэтому правильный ответ — Docker с
   `restart: unless-stopped` (чанк ещё не собран).

10. **Молчаливое падение хендлера.** Есть гарды на `ValueError` (устаревшие
    карточки), но ошибка БД/Telegram API не ловится — юзер видит вечный
    спиннер без ответа. Нужен глобальный `@dp.errors` хендлер: залогировать +
    «что-то пошло не так, попробуй ещё раз».

11. **FSM в памяти** — незавершённый ввод «своей категории» теряется при
    рестарте. Мелочь (потеряется только один клик), но упоминаю.

## E. Качество кода / производительность (низкий приоритет для 2 юзеров)

12. **N+1 запросы.** `_refresh_weights`, `next_questions`,
    `find_regular_payments` делают по запросу на контрагента. На 300 операциях
    незаметно, но грязно. Переписать на агрегатные запросы, когда дойдут руки.

13. **Самописные ALTER TABLE миграции** ([db.py](../finbot/db.py)) — хрупко по
    мере роста схемы. «Правильный» ответ — Alembic, но для спеки это перебор.

14. **Нет тестов на хендлеры `bot.py`.** Вся логика покрыта, но баги A.1–A.3
    живут именно в непокрытом слое. Стоит добавить тесты через тестовый
    Dispatcher/mock.

## F. Робастность парсера

15. **Проверен на одной реальной RU-выписке.** Зрелый проект
    `sebastienrousseau/bankstatementparser` независимо использует те же
    приёмы, что и мы (Golden Rule, дедуп, continuity-проверка) — это хорошая
    валидация нашего подхода. Но он же покрывает мультивалютность, сканы,
    page-provenance. Наш USD-continuation захардкожен под один паттерн, другие
    продукты Kaspi (Red, Депозит) не проверены. Golden Rule страхует (откажет,
    а не соврёт), но покрытие тонкое. Нужны реальные KZ/EN/длинные выписки.

## G. Осознанные расхождения со спекой (НЕ баги — не «чинить» обратно)

- **Окно неттинга §5.6:** спека ±2 дня, мы расширили до 7 (покупки) / 30
  (переводы между людьми) по твоему фидбеку (долг «закинул — вернул через
  неделю»). Задокументировано намеренно.
- **Частичные возвраты:** плюс на расходном контрагенте вычитается из его
  категории, а не идёт в «доход» — за пределами буквы спеки, по твоему кейсу
  (Smile: −40к покупка + 4к возврат = −36к).

## Рекомендуемый порядок работ

1. Блок A (3 бага корректности) — быстро и важно.
2. Пункт 8 (бэкап/`/export`) — дёшево, снимает главный риск потери данных.
3. Пункт 10 (глобальный error-handler) — одним махом убирает «зависания».
4. Блок B (исправление ответов, произвольный период, управление категориями)
   — по одному, это то, что делает бота пригодным на месяцы.
5. Чанк Docker (пункт 9) — чтобы бот жил без ручного перезапуска.
6. Остальное (E, F) — полировка.
